## Context

SpecStat is a brand-new product with no prior implementation. The challenge is building a rich visualization UI that operates with zero backend infrastructure — GitHub itself acts as the data store, and GitHub API is the only data access layer. The codebase lives in a monorepo (`specstat/`) structured as a Turborepo with a Next.js web app and shared packages.

The critical constraint is GitHub API rate limits (5,000 req/hr for authenticated users). All data access must be lazy and aggressively cached.

## Goals / Non-Goals

**Goals:**
- Zero-backend architecture — GitHub is the only data source
- Non-destructive integration — OpenSpec files are never modified, only `visualize.json` files are added alongside
- Two-way triggers — Visualizer UI can dispatch GitHub Actions; Markdown files can embed trigger directives
- Multi-repo workspace — unified view across multiple repos
- Offline-friendly — stale-while-revalidate via TanStack Query

**Non-Goals:**
- Real-time collaboration or live cursors
- In-app spec editing (read + status-drag only; full edits stay in GitHub)
- Self-hosted deployment (Vercel-only for now)
- Support for non-GitHub hosting (GitLab, Bitbucket)
- Custom role-based access control (defers to GitHub permissions)

## Decisions

### D1: Distributed `visualize.json` over a centralized database
Each spec item and folder owns its own `visualize.json` alongside the spec file. The `index.json` at repo root is a flat manifest generated by GitHub Actions.

**Alternatives considered:**
- Single large JSON blob — fails at scale, causes merge conflicts
- Centralized database (Supabase, PlanetScale) — introduces backend dependency and ops burden

**Rationale:** Distributed ownership matches git semantics. Each file change touches only the relevant `visualize.json`. The index is regenerated by CI, not by hand.

---

### D2: GitHub Actions for all write operations
All mutations to `visualize.json` and `index.json` happen through GitHub Actions, not through the Visualizer frontend directly writing files.

**Exception:** Status changes triggered by card drag-and-drop in the UI call the GitHub API directly (via `github-client/updateVisualizeJson.ts`) to commit a single file change, since this is a lightweight targeted update.

**Alternatives considered:**
- Visualizer backend writing files — adds infrastructure
- PR-based edits only — too slow for status drag/drop UX

**Rationale:** Actions handle bulk operations (init, sync, baseline). Direct API commits handle atomic single-file mutations (status change).

---

### D3: Next.js App Router with TanStack Query for data fetching
Next.js 14 App Router for routing and Server Components where beneficial. TanStack Query (React Query) for all client-side GitHub API data fetching with stale-while-revalidate caching.

**Rationale:** TanStack Query's cache semantics (staleTime, refetchOnWindowFocus, optimistic updates) are a natural fit for GitHub API data that changes infrequently. Server Components reduce JS bundle for static content.

---

### D4: Monorepo with Turborepo
`apps/web` for the Next.js app. `packages/openspec-parser`, `packages/github-client`, `packages/openspec-action`, `packages/ui`, `packages/types` as shared packages.

**Rationale:** The GitHub Action (`openspec-action`) shares parsing logic with the frontend. Turborepo build caching prevents redundant builds.

---

### D5: React Flow for graph view
React Flow handles node/edge rendering, zoom, pan, and selection for the relationship graph.

**Alternatives considered:**
- D3.js — lower-level, more custom work
- Cytoscape.js — heavier, less React-native

**Rationale:** React Flow is React-native, has built-in node/edge types, and handles large graphs with virtualization.

---

### D6: dnd-kit for drag-and-drop board
dnd-kit for the Jira-like board drag-and-drop between status columns.

**Alternatives considered:**
- react-beautiful-dnd — unmaintained
- @hello-pangea/dnd — fork but less active

**Rationale:** dnd-kit is actively maintained, accessible, and works with React 18.

---

### D7: next-auth for GitHub OAuth
next-auth v5 handles GitHub OAuth sign-in, session management, and token storage.

**Rationale:** next-auth abstracts OAuth flow, CSRF protection, and session cookies. GitHub provider is first-class supported.

---

### D8: Zod for runtime schema validation
All `visualize.json` files are validated with Zod schemas on parse. The same Zod schemas generate TypeScript types.

**Rationale:** Single source of truth for schema — types and validation derived from the same definition. Errors surface early before rendering.

## Risks / Trade-offs

- **GitHub API rate limits** → Mitigation: TanStack Query with 5-minute staleTime, request deduplication, lazy loading (fetch items on demand from index, not all at once)
- **Large repos with 1000+ spec items** → Mitigation: Virtual scrolling in board/tree views, pagination on timeline
- **Stale index.json** → Mitigation: Sync Action runs on every push to `openspec/**`; Visualizer shows `last_updated` timestamp prominently
- **GitHub API latency on cold load** → Mitigation: index.json is the only mandatory first fetch; all item details are lazy
- **Status drag-and-drop commit race** → Mitigation: Optimistic UI update + background commit; conflict on concurrent edits surfaces as a GitHub API error shown in toast
- **Multi-repo cross-ref resolution** → Mitigation: Each index carries its `repo` field; cross-repo links open in a new workspace panel rather than inline resolution

## Migration Plan

1. No migration needed — additive convention on new repos
2. Existing OpenSpec repos: run `openspec-init` Action once manually; it generates all `visualize.json` and `index.json` without touching existing files
3. Rollback: delete all `visualize.json` files and `index.json` — no OpenSpec files are modified so nothing is lost

## Open Questions

- Should status drag-and-drop create a commit directly on the default branch or open a PR? (Leaning toward direct commit for low-friction UX; team leads can restrict with branch protection)
- GitHub OAuth app: single shared SpecStat OAuth app, or users bring their own GitHub token? (Leaning toward shared OAuth app for simplicity)
- Should `openspec-action` be published as a Docker container action or a Node.js action? (Node.js is faster — no container pull)
